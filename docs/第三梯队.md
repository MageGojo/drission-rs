## 里程碑 47 · 吐环境指纹回放深化(④)

### 目标
里程碑 31 已覆盖 **canvas(toDataURL)/ webgl / audio** 的录制+Node 回放+双跑验证。本里程碑补齐
进度表「后续可增强 #1」列出的四个高强度指纹点,使补环境 `env.js` 在 Node 侧能忠实回放:

1. **字体枚举**(font enumeration):`measureText` 宽度法探测已装字体。
2. **像素级 canvas**(`getImageData`):读取像素并哈希(很多指纹库读 `getImageData` 而非 `toDataURL`)。
3. **WebRTC**:`RTCPeerConnection` 是否可用 + 编解码能力概要(默认 `block_webrtc=true` 时即 `supported:false`)。
4. **`navigator.plugins` / `navigator.mimeTypes`**:插件/ MIME 列表(常被读做指纹)。

### 关键设计:录制==回放「单一配方」(沿用里程碑 31)
- 新增配方函数放进 `assets/fp_recipes.js`(探针采集 seed 与 verify 快照**共用同一份**):
  - `__fpFonts()` → `{ supported, detected:[...], widths:{"<font shorthand>":w}, hash }`
  - `__fpCanvasPixels()` → `{ supported, width, height, hash, data }`(`data` = `getImageData` 字节的 base64,供回放)
  - `__fpRtc()` → `{ supported, codecsHash }`
- 配方只依赖 `document.createElement('canvas').getContext('2d')` / `typeof RTCPeerConnection`:
  - **浏览器**里走真实实现 → 真实指纹(录制)。
  - **Node** 里走 `env.js` 注入的假 DOM(回放):
    - `measureText` 返回**录制的宽度**(按 `ctx.font` 串查表)→ `__fpFonts()` 复算出同样的 detected/hash。
    - `getImageData` 返回**录制的字节**(从 `seed.fingerprint.canvasPixels.data` 还原)→ 同 hash。
    - `RTCPeerConnection` 缺省不存在 → `supported:false`(与默认 `block_webrtc` 浏览器侧一致)。

### 录制(探针 `collectSeed`)
- `seed.fingerprint` 增 `fonts` / `canvasPixels` / `rtc`(均同步配方,不走 audio 的异步路径)。
- `seed.navigator` 增 `plugins` / `mimeTypes`(数组,元素含 name/filename/description、type/suffixes 等)。

### 回放(`env_template.js`)
- `makeCtx2D`:`measureText` 改为**查录制宽度**(`__SEED__.fingerprint.fonts.widths[this.font]`,缺失回退旧估算);
  `getImageData` 改为**还原录制字节**(`canvasPixels.data` base64 → `Uint8ClampedArray`,尺寸匹配时用,否则回退 0)。
- `RTCPeerConnection`:`supported:true` 才 stub 一个最小构造器 + `RTCRtpReceiver.getCapabilities`,否则**保持 undefined**。
- `navigator.plugins/mimeTypes`:`setup` 后把 seed 数组包成**类数组**(`length` + 下标 + `item`/`namedItem`/`refresh`),
  并交叉链接 `mimeTypes[i].enabledPlugin`。

### 验证(`dump_env.rs` 双跑 + 导出工程 `verify.js`)
- `gen_snapshot_js`:新增比对项 `fonts.hash`/`fonts.count`、`canvasPixels.hash`、`rtc.supported`、
  `navigator.pluginsCount`/`navigator.plugins0`/`navigator.mimeTypesCount`(后三者浏览器/Node 实时双读)。
- `fill_recorded_fp`:`fonts.hash`/`canvasPixels.hash`/`rtc.*` 用 seed 录制值(canvas 类指纹 Camoufox 可能逐次加噪,
  比对「录制 vs Node 回放」而非「实时 vs 回放」,与 canvas.dataURL 同策略)。
- 导出工程 `verify.js`:增字体/像素/rtc/plugins 的回放自检项。

### 取舍 / 边界
- **footprint**:`canvasPixels` 用紧凑画布(96×32 = 12288 字节 → base64 ~16KB)控制 `env.js` 体积。
- **WebRTC**:默认 `block_webrtc=true` 下 `supported:false`,回放即「RTCPeerConnection 仍 undefined」——这正是要补的一致性。
- **plugins 类型**:回放是类数组(`length`/下标/`namedItem`)而非真 `PluginArray`;`Object.prototype.toString` 仍为 `[object Object]`,
  对绝大多数 `plugins.length`/`plugins[i].name` 读法已足够(进一步伪装可后续按目标站点叠加)。
- 不引入任何新依赖;占位符替换沿用「恰好 1 次」单测兜底(里程碑 20 教训)。

## 里程碑 54 · 点选 / 文字点选验证码(①)

### 目标
对标 ddddocr `det=True`,补齐验证码最后一块:**目标检测 + 逐框 OCR + 顺序匹配**。案例 = 易盾
`https://dun.163.com/trial/picture-click`(雪山背景上若干艺术体汉字,提示"请依次点击 X Y Z")。

### 实现(`src/ocr/mod.rs`,`feature = "ocr"`,后端无关)
- **`Det`(检测器)**= ddddocr `common_det.onnx`(**YOLOX**),**tract 纯 Rust 推理**(沿用识别同款引擎,
  无原生 onnxruntime)。流水线严格对齐 ddddocr:416×416 灰边(114)letterbox、**原始 0–255 RGB 不归一化**、
  NCHW → 解码 `(xy+grid)*stride` / `wh=exp*stride` → 分数阈值 0.1 → NMS 0.45 → 还原原图坐标。
  `Det::new()`(首次下载 `common_det.onnx` 到缓存)/ `from_model_path` / `detect(&[u8]) -> Vec<BBox>`。
- **`BBox`**(原图像素 + score + `center()`)。
- **`ClickWord`(点选求解器)**:`chars(img)`=检测 → 逐框裁剪(适度外扩)→ `Ocr::recognize` 逐框识别 →
  `Vec<(BBox, 文本)>`;`points_for(img, targets)`=按提示顺序匹配(文本相等 → 互相包含,框不复用)→ 依次点击点。
  顺序匹配抽为纯函数 `match_order` 便于单测。
- 导出 `BBox`/`Det`/`ClickWord` 进 `prelude`(ocr feature)。

### 关键结论
- **tract 能跑 ddddocr 检测模型**(YOLOX 纯卷积,不像旧识别 `common_old.onnx` 的 `DynamicQuantizeLSTM` 不支持)——
  `det_probe` 实测加载/优化/推理全通。
- 易盾验证码图是 **CSS 背景图**(`.yidun_bgimg` DIV,无 `<img src>`)→ 取像素须**元素截图**;挑战需**可信点击**
  `.yidun_control` 才弹出(JS `.click()` 不认);提示是明文 `.yidun_tips__text`("请依次点击 …")可直接解析顺序。

### 验证
- 库单测:`det_grids_count_matches_yolox`(3549=52²+26²+13²)、`match_order_follows_targets_and_skips_missing`(顺序/跳过/不复用);`--features ocr` lib 全过、clippy 新代码零警告。
- `examples/det_probe`(CDP 截图喂 Det,验证 tract 可行)+ `examples/yidun_click`(易盾全链路:触发 → 可信点击控件 →
  截验证码图 → 检测/识别 → 解析提示 → 算点击点 → 可信点击)。
- **易盾实测**:检测 4~5 框置信 **0.85–0.88**、提示解析「元/验/体」、逐框 OCR `验`/`体` 命中、顺序匹配出点击点。

### 局限(已知)
- **单字艺术体 OCR 非 100%**:ddddocr 识别模型针对常见验证码字体,易盾艺术/书法字偶有误识(如 `元`→`带`),
  整体命中率受此限;需更高命中可换 dddd_trainer 自训模型(里程碑 ③ 计划:模型热替换)。
- **行为风控**:与极验/顶象同理,"字点准 ≠ 必过",易盾另有轨迹/指纹/IP 风控层(与识别算法两件事)。

## 里程碑 55 · 点选**置信度 + 多候选纠偏**强化(①续)

### 背景
里程碑 54 把点选链路打通(易盾实测坐标正确、命中 3/3 可信点击),但易盾**换图未过**。两因:① 艺术体单字
OCR 非 100%;② 行为风控。用户要求先把**当前点选能力(检测+识别+顺序)的置信度 / 多候选纠偏再调强**(②③ 后续)。

### 改动(`src/ocr/mod.rs`,后端无关,零新依赖)
- **指派算法**:`ClickWord` 的"框 ↔ 目标字"配对从 **按目标序贪心** → **全局最优指派** `assign_optimal`
  (纯函数,**DFS + 分支定界**):每个目标分到**互不相同**的框、**最大化总亲和度**;框够时每个目标都分到框
  (`ASSIGN_BONUS` 让"多分配一个"优先于"亲和度更高"),框不足时由总亲和度决定哪些目标空缺。
  - 贪心是次优:靠前目标抢走靠后目标更需要的框(`[[.9,.8],[.85,.1]]` → 贪心 1.0,全局 1.65)。
- **置信度 API**:新增 `ClickHit{target, bbox, point, affinity}` + `ClickWord::solve()`(逐目标命中 + 亲和度);
  `points_for` 退化为 `solve` 取点;`points_for_text`(文本 argmax 法)保留作回退。`ClickHit` 进 `prelude`(ocr)。

### 验证
- 3 个新纯函数单测:全局指派**胜过贪心** / 框够时**互异** / 框不足**部分指派**(+ 空边界)。`--features ocr` lib **71** 全过。
- `clippy --lib --examples --features cdp,ocr` 干净;默认构建(无 ocr)不回归。
- 示例 `yidun_click`:改用 `solve` 打印**逐字置信度** + **最低置信度门控**(`YIDUN_MIN_CONF`,默认 0.30)+
  **换图重试**(`YIDUN_TRIES`,默认 3,点 `.yidun_refresh`);提示含"成功"即停。

### 边界(不变)
- 艺术字单字 OCR 命中率上限属 ddddocr 固有 → 真要拉满需 **③ 模型热替换**(dddd_trainer 自训)。
- 行为风控(轨迹/指纹/IP)与识别是两件事 → **② 滑块/点选行为轨迹模型化**。

## 里程碑 56 · 点选识别增强(多信号路线,①续续)

用户:"忘掉不能点选,想办法给库加上点选功能"。既然检测+全局指派已稳、瓶颈在**艺术字单字 OCR**,
按"多信号融合"分阶段攻这一步(都在 `feature=ocr`):

- **阶段1(✅ 本里程碑)·逐字多预处理 TTA**:`glyph_variants` = `[原图, 自动对比度, Otsu 去背景]`,
  `ClickWord::solve` 逐变体识别 + `select_by_margin` 按锐度(top1-top2)选最佳整向量。**纯 `image`、零新依赖、
  只增不减**。实现用户"把背景暗掉让字浮出"的思路。有头实跑:同图旧版多字 0.00–0.17 → 新版 `特0.81/扩0.99/体0.74`
  全高置信放行点击(仍未过 = 艺术字确信误读 + 行为风控)。
- **阶段2(✅ 里程碑 58)·字形模板匹配融合**:提示已给标准答案字 → **自渲染**(系统 CJK 字体,`fontdue`
  纯 Rust,`DRISSION_CJK_FONT` 可指)为模板,与各框做**形状相似度**(`N×N` 零均值 L2 归一向量点积=NCC;
  模板多旋转取最大、字框用 Sobel 梯度幅值),`combo = OCR + 1.5×模板` 再全局指派 —— 专治"OCR 确信误读"。
  `src/ocr/glyph.rs` 的 `GlyphMatcher`,`ClickHit.template` 暴露子分。**实跑**:某题 OCR 全 0.00 时模板仍给
  `扩0.36/类0.48/元0.24` 独力支撑指派(OCR 失效时优雅降级);最难艺术字两信号都弱 → 仍需阶段3。
- **阶段3·模型热替换**:`Ocr`/`Det` 支持加载自定义 `.onnx`(`from_model_path` 已有,补**运行时热替换** API +
  dddd_trainer 训练流程文档),用易盾艺术字自训模型 —— 精度天花板。
- **输入侧增强(随手)**:拦截/直取验证码**原图高分辨率字节**(而非 320px 视口截图)喂识别,像素更多更准。

# 设计文档

## 总体架构(自顶向下分层)

```
┌──────────────────────────────────────────────────────────────┐
│  高层 API(DP 风格)  browser/                                  │
│   Browser ── Tab(标签/页面) ── Element(元素)                  │
│   Tab.listen(网络监听) / Tab.cookies(每标签 cookie)           │
├──────────────────────────────────────────────────────────────┤
│  协议层  protocol/                                              │
│   Connection(请求↔响应配对、事件分发) / Session(root & page)  │
│   message(线消息类型) / methods(66 方法 + 33 事件强类型封装)  │
├──────────────────────────────────────────────────────────────┤
│  传输层  transport/                                             │
│   PipeTransport(异步包裹子进程 fd3 写 / fd4 读)               │
│   codec(null 分隔 JSON 帧编解码)                              │
├──────────────────────────────────────────────────────────────┤
│  启动器  launcher/                                              │
│   options(选项 / 指纹) / fetch(自动下载分发) / process(spawn)│
└──────────────────────────────────────────────────────────────┘
              │ 子进程 + fd3/fd4 管道(Juggler)
              ▼
        Camoufox(Firefox 内核 + 反指纹)
```

设计原则:**组件化**,每层职责单一、可独立测试;协议层与传输层不感知高层语义。

## 为什么是 Juggler(而不是 CDP)
- Camoufox 是 Firefox 分支,自动化协议是 **Juggler**(Playwright 的 Firefox 协议),
  经由命令行 `-juggler-pipe` 在 **fd3(读命令)/ fd4(写响应)** 上通信。
- 帧格式:**UTF-8 JSON + 单个 `\0`**;连续 `\0` 为空帧需忽略;消息可跨多次 read。
- 会话模型:**root 会话**(无 `sessionId`,处理 `Browser.*`)+ 多个 **page 会话**
  (带 UUID `sessionId`,处理 `Page.*/Runtime.*/Network.*/Heap.*`)。

## 异步与并发模型
- 运行时:`tokio`(multi-thread)。
- 连接内有一个**读任务**:循环读 fd4 → `codec` 成帧 → 解析 → 按 `id` 唤醒等待者 /
  按 `sessionId` 分发事件。
- 每个请求分配全局自增 `id`,用 `oneshot` 等待响应;每个会话/标签有独立事件广播。
- 多标签并发:不同 page 会话彼此独立,可在不同 `tokio` 任务里并发驱动。
- cookie 隔离:每个标签放在**独立的 BrowserContext**(`removeOnDetach: true`),
  从而 `Browser.getCookies/setCookies(browserContextId)` 实现"每标签不同 cookie"。

## 启动引导序列(来自协议规范)
1. spawn:`camoufox -no-remote [-headless|-foreground] -profile <dir> -juggler-pipe [-silent]`
   stdio = `[ignore, pipe, pipe, pipe, pipe]`(fd3 写命令,fd4 读响应)。
2. 监听 **stderr** 出现 `"Juggler listening to the pipe"`(超时 180s)。
3. 并行:`Browser.enable` + `Browser.getInfo`(+ 代理)。
4. `Browser.createBrowserContext({removeOnDetach:true})` → `browserContextId`。
5. 并行下发上下文级覆盖:`setUserAgentOverride / setLocaleOverride / setTimezoneOverride /
   setGeolocationOverride / setDefaultViewport / setBypassCSP / setIgnoreHTTPSErrors / setContextProxy / setInitScripts …`。
6. `Browser.newPage({browserContextId})` → 等 `Browser.attachedToTarget` 拿 `sessionId`。
7. 在 page 会话上等 `Page.frameAttached`(父帧为空 → 主帧 `frameId`)与
   `Runtime.executionContextCreated`(主世界 `name==""`)拿 `executionContextId`。

## 元素定位与操作流程
- 定位:`locator::parse` 把 DP 选择器解析成 `Css(..)` 或 `Xpath(..)`。
- 查找:`Runtime.evaluate` 跑 `document.querySelector` 或 `document.evaluate(xpath)`,
  返回 `RemoteObject{subtype:"node", objectId}`。
- 点击:`Page.scrollIntoViewIfNeeded` → `Page.getContentQuads` 取中心点 →
  `Page.dispatchMouseEvent`(move/down/up)。
- 输入:聚焦后 `Page.insertText`(或逐字符 `Page.dispatchKeyEvent`,回车走 `\r`)。
- 读属性/文本:`Runtime.callFunction` 在节点上取 `innerText/getAttribute(...)`。
- 填表:`<select>` 用 `select_value/select_text`(置 value + 派发 change);checkbox/radio 用真实
  `click` 或 `set_checked`/`is_checked`;多步表单靠点击"下一步/提交"按钮推进。

## 网络监听 / 拦截(对应 DP 的 tab.listen + 增强)
- 监听:`Network` 事件(`requestWillBeSent / responseReceived / requestFinished`),
  按 URL 子串 + 资源类型(`xhr/fetch/...`)过滤;`Network.getResponseBody` 取响应体。
  注意:Camoufox 实测发原始类型名(如 `TYPE_XMLHTTPREQUEST`),大小写不敏感子串匹配。
- 拦截:page 会话 `Network.setRequestInterception{enabled:true}` 后,被拦请求在
  `requestWillBeSent` 带 `isIntercepted:true` 暂停(Juggler **无独立拦截事件**)。
  库策略:**匹配过滤**的请求投递给 `intercept_next` 交用户决策(`resume`/`resume_with`/
  `fulfill`/`abort`),**不匹配**的由事件泵自动 `resumeInterceptedRequest` 放行,避免卡死。

## 控制台监听(对应 DP 的 tab.console,里程碑 25)
- **走原生事件**:与网络监听用页面 hook 不同,控制台监听订阅 Juggler **原生 `Runtime.console` 事件**
  (默认就转发到 page 会话,无需 `Runtime.enable`)——**不污染页面 `console` 对象**,对反检测库更友好。
  事件:`{executionContextId, args:RemoteObject[], type, location:{url,lineNumber,columnNumber}}`(行列 0 基)。
- **文本还原(混合策略)**:RemoteObject 里**原始类型**(字符串/数字/布尔/`NaN`)直接带 `value`,在 Rust 侧
  零往返拼 `text`;**对象/数组/Error/DOM 节点**只给 `objectId`(Juggler 无 preview),回调一次
  `Runtime.callFunction`(把事件 RemoteObject 削成 `CallFunctionArgument` 的 objectId/value/unserializableValue)
  在页面里 `JSON.stringify`/`String` 序列化后拼接——故 `console.log({a:1})` 的 `text` 即 `{"a":1}`,`body()` 可解析。
- **独立后台任务**:`console.start()` 起一个订阅任务(自己按 session 过滤),对象序列化的 `callFunction` 在该任务
  里 await,**不阻塞主事件泵**;缓冲为有上限 `VecDeque`(超限丢最旧)。`wait` 轮询缓冲(与库内其它等待一致)。
- **来源与过滤**:Juggler 把 console API(log/info/warn→warning/error/debug/…)与 console-service(页面告警)
  并到同一事件、无法在线区分,故 `source` 统一 `console-api`(best-effort);另给 drission 增强的
  `ConsoleFilter`(级别 / `text` 子串)在任务侧过滤噪声(DP 的 `console.start()` 无过滤)。

## 过盾 / 拟人化接线
- `humanize`:序列化进 `CAMOU_CONFIG_1`(`{"humanize":true,"humanize:maxTime":t,"showcursor":false}`),
  浏览器侧 `MaskConfig` 读 `CAMOU_CONFIG_1..n` 拼接;影响经 Juggler 下发的鼠标移动轨迹。
- `block_webrtc` / 自定义 prefs:Firefox user prefs(如 `media.peerconnection.enabled=false`),
  经 `Browser.enable{userPrefs:[{name,value}]}` 下发。
- 基础硬化:Camoufox 二进制自带(headless 也打补丁);`navigator.webdriver=false`,
  实测可自动通过真实 Cloudflare JS challenge。

## 错误处理
- 统一 `Error`/`Result`;协议错误取服务端 `error.message`;`sendMayFail` 类方法吞错不传播。

## 基础页面能力(对标 DP,里程碑 21)
- **健壮 `get`**:`Result<bool>`(失败返回 `false` 不抛错)+ `GetOptions`(retry/interval/timeout/load_mode/referer)。
  `LoadMode::{Normal,Eager,None}` → 等 `load` / `DOMContentLoaded` / 不等(事件名取自 Juggler `Page.eventFired`);超时再退查 `readyState` 兜底。
- **页面状态**:`user_agent`/`ready_state`/`url_available`(同步读 `last_load_ok`)/`stop_loading`(JS `window.stop()`)/`wait_loaded`。
- **静态元素**:`tab.s_ele/s_eles/s_root` 取当前 HTML,用 `scraper`(html5ever)离线解析为 `StaticElement`(`Rc<Html>`+`NodeId`,非 `Send`);
  DP 选择器经 `locator::parse_static` 落到 CSS / 属性 / 文本直接匹配,显式 `xpath:` 走内置 `xpath.rs` 子集求值器。
- **句柄对象**:`tab.wait()/scroll()/set()/listen()` 返回轻量句柄(各持一个 `Tab` 克隆),把相关操作收敛成 `tab.wait.*`/`tab.scroll.*`/`tab.set.*`/`tab.listen.*`。

## 锦上添花四项(对标 DP,里程碑 22)
- **iframe 内元素**:事件泵维护 `frame_ctxs`(每个子帧的主世界 `executionContextId`,来自 `executionContextCreated.auxData.frameId`);`tab.get_frame(sel)` 经 `Page.describeNode→contentFrameId` 得 `Frame`,`Frame`/帧内 `Element` 在该帧 ctx 内 evaluate/callFunction。
- **文件上传 / 对话框**:`ele.set_files/upload` 走 `Page.setFileInputFiles`;`tab.handle_next_dialog(accept, prompt)` 订阅 `Page.dialogOpened` 并回 `Page.handleDialog`(返回 `DialogInfo`)。对话框阻塞 JS,触发与处理需 `tokio::join!` 并发。
- **静态 XPath**:`xpath.rs` 自研 XPath 1.0 子集(路径/谓词/`contains`/`text()`/`and/or/not`),位置谓词按同父同名兄弟计数;不支持的轴报错提示改用实时 `ele`(浏览器原生 `document.evaluate`)。
- **`tab.listen()` 句柄**:`Listen`(handles.rs)轻量包裹 `Tab`,`start/wait/wait_count/steps/stop` 全部转调既有扁平 `listen_*` 实现,行为不变。

## 相对定位与 Shadow DOM(里程碑 41,对标 DP)

- **相对定位**(`Element` 上的一组方法):`parent`/`parent_n(level)`/`parent_until(loc)`、`children`/`child(i)`、
  `next`/`prev`/`nexts`/`prevs`/`siblings`。实现走 `Runtime.callFunction` 在元素所属 frame 上下文里
  遍历 DOM 关系(`parentElement`/`children`/`nextElementSibling`…),返回的 `Element` **继承锚点的 `frame_id`**
  (iframe 内相对定位也正确)。返回单个的找不到统一 `Err(ElementNotFound)`(与 `ele` 一致);`child(i)` 用
  **0 基**(与本库 `get_tab` 等索引一致,异于 DP 的 1 基)。`parent_until` 用 `closest(css)`,仅 CSS 系定位。
- **Shadow DOM**(`ShadowRoot`,`src/browser/shadow.rs`):`ele.shadow_root()` 取 `node.shadowRoot`(open 才非 null),
  返回一个**查找上下文**(非元素,故无 click/text)。`root.ele/eles` 在 shadow root 上 `querySelector(All)`——
  shadow 内**无 `document.evaluate`**,故只支持 CSS 系定位(`xpath:` 报错)。shadow 内查到的仍是普通 `Element`,
  归属同一 frame(shadow 不创建新执行上下文),后续操作照常。
- **底层去重**:`Tab::eles`/`Frame::eles`/相对定位/`ShadowRoot::eles` 共用 `TabCore::node_array_object_ids`
  (把"节点数组 RemoteObject"经 `Runtime.getObjectProperties` 展开为各节点 `objectId`),消除原先 tab/frame 两处
  重复的展开逻辑。端到端自验证见 `examples/relative_shadow`(本地 `file://`,完全离线)。

## 拖拽能力(里程碑 23)
- **低层鼠标**:`tab.mouse_move`(buttons=0)/`mouse_down`/`mouse_drag`(buttons=1,拖拽中的 move)/`mouse_up`,统一经 `TabCore::dispatch_mouse(type,x,y,buttons)` 下发 `Page.dispatchMouseEvent`(可信事件,能真正驱动滑块)。
- **元素拖拽**:`ele.drag(dx,dy,dur)`/`drag_to(x,y,dur)`/`hover()`;轨迹由 `human_drag_track`(element.rs,std-only xorshift)生成"ease-out + 过冲回拉 + y 抖动 + 随机停顿",末步精确落点;`dur` 缩放各步停顿到目标总时长。
- **滑块验证码闭环**(`examples/geetest_slide`):读 gt4 三张 canvas 像素算缺口(`bg` vs `fullbg` diff,跳过左侧拼图块),`mouse_down`→多次 `mouse_drag`(其间 `run_js` 读拼图块真实位置)→`mouse_up` 闭环纠偏到 ≤1px。geetest 行为风控仍判 headless 失败(其反爬强项,非拖拽问题)。

## 拟人输入深化(里程碑 32,由 GeeTest 诊断驱动)
- **诊断先行**(`examples/geetest_probe`):在滑块把手 + `document` 捕获阶段挂事件记录器,实测极验从我们 `Page.dispatchMouseEvent` 收到的事件——`isTrusted=true`、`pointer*`+`mouse*` 双发、`movementX/Y` 真实、`screenX≠clientX`、`pressure` 0.5/0、环境干净;定位两处破绽:`pointerType=""`(应 `"mouse"`)与采样 ~60ms/点(应 ~10~16ms)。
- **fire 快路径**(密集采样):`Connection::fire_session`(不等往返、带 session;id 从内部计数器分配,回包查无 pending 在 `dispatch` 处安全丢弃)→ `TabCore::dispatch_mouse_fire` → `Tab::mouse_move_fast/mouse_drag_fast`。把"事件间隔"交给 `sleep`,绕开每事件 ~20ms 往返,实测采样 ~60ms/10点 → ~20ms/32点。**闭环屏障**:fire 后读位置前用一次会等待的 `mouse_drag`(FIFO,响应回来即代表前面 fire 全处理完)。
- **minimum-jerk 轨迹**:`human_drag_track` 改为神经运动学 minimum-jerk 速度 `10t³-15t⁴+6t⁵`(钟形)+ 时间驱动密集采样 + ±40% 延时抖动 + 手抖 + 纵向漂移 + 偶发迟疑 + 过冲回拉;`ele.drag` 中间 move 走 fire 快路径。
- **pointerType 反检测**:`Tab::apply_pointer_stealth()` 导航前注入 getter 补丁,把空 `pointerType` 补成 `"mouse"`(只补一处、不动 `Function.prototype.toString`,避免引入更大可扫描面)。
- **边界**:极验 v4 行为 ML + 会话信任分 + 可能的 `getCoalescedEvents()` 检查属"另一量级",输入层已做到位仍会被判失败——这是 captcha 对抗本身的难度,与库能力无关。

## 截图与录像(对标 DP `browser_control/screen`,里程碑 27)

设计原则:**大道至简**——常用 80% 走最短方法,余下能力收进一个 options / 句柄,不堆方法。

- **图片格式 `ImageFormat`**:仅 `Png`/`Jpeg`(Camoufox 的 `Page.screenshot` 实测只认 `image/png`/`image/jpeg`,
  不谎称 webp)。`from_path` 按扩展名推断(`.jpg`/`.jpeg`→Jpeg,其余→Png),`get_screenshot(path,..)` 据此自动选格式。
- **底层一处** `TabCore::capture(clip, format, quality)`:发 `Page.screenshot{mimeType, clip, quality?}` → base64 解码字节。
  裁剪区计算也收进 `TabCore::page_clip(full_page)`(整页用 `scrollWidth/Height`、视口用 `scroll+inner`,与旧实现一致)。
- **页面截图**(`Tab`):
  - `get_screenshot(path, full_page) -> PathBuf`(保留;格式按 path 后缀)、`screenshot_bytes(full_page) -> Vec<u8>`(保留;PNG)。
  - `screenshot_base64(full_page) -> String`(对应 DP `as_base64`)。
  - `screenshot(&ShotOpts) -> Vec<u8>`(全能力:`full_page` / `region(左上,右下)`(对应 DP `left_top`/`right_bottom`)/ `format` / `quality`)。
- **元素截图**(`Element`):`get_screenshot(path)` / `screenshot_bytes()`——**先把元素滚到视口中央**(DP 默认 `scroll_to_center=true`),
  再取 `getBoundingClientRect + scrollX/Y` 作为页面坐标裁剪区截图。
- **录像 `tab.screencast()` 句柄**(对标 DP `tab.screencast`):**大道至简 = 后台任务按帧间隔反复截视口**(复用 `capture`),
  不走 Juggler 原生 `startScreencast`(那要处理 `screencastFrame`/`ack`/节流,复杂度更高)。
  - 模式 `ScreencastMode`:`Imgs`(持续逐帧存图)/`FrugalImgs`(画面变化才存,按字节比对去重)/`Video`/`FrugalVideo`
    (取帧后停止时用 **ffmpeg** 合成 mp4;无 ffmpeg 则报错指引——对标 DP 的 video 模式需 opencv,这里换成更通用的 ffmpeg CLI,**不引 Rust 依赖**)。

## WS 接管浏览器(Browser::connect + BrowserServer,里程碑 29)

对标 DrissionPage「接管已运行的浏览器」。当前 Camoufox 二进制**已移除**浏览器内置 `-juggler <port>`(ws)模式
(实测裸传报 `unrecognized command line flag`,只剩 `-juggler-pipe`),且 `camoufox server` 暴露的是 Playwright RPC
协议(非原始 Juggler,我们的 Juggler 客户端讲不了);故采用**自建中转**。

- **传输层抽象**:`Connection` 不再绑死 fd3/4,拆为两个构造器——
  - `from_pipe(writer, reader)`:本地启动场景,帧格式 `<JSON>\0`,写任务加分隔符、读任务 `FrameDecoder` 拆帧;
  - `from_ws(WebSocketStream<S>)`:接管场景,**一条 ws 文本消息 = 一条完整 Juggler JSON**(ws 自带消息边界,无需 `\0`);
  - `send/fire` 改为投递**未成帧的原始 JSON** 到命令队列,成帧动作**下沉**到各传输写任务;`dispatch`/请求响应配对/事件广播完全共用。
- **服务端 `BrowserServer`**(`browser/serve.rs`):`launch[_on]` 启动 Camoufox(走管道)+ 起 `TcpListener` 做 ws 监听,
  把管道帧与 ws 客户端做"消息级"双向转发(单管道写任务 + 单管道读任务 `broadcast` + 每客户端两个泵),`ws_endpoint()` 返回 `ws://host:port/token`。
- **单活动客户端**:原始 Juggler 是单逻辑连接、单自增 id 空间,多个 ws 客户端复用同一管道会 id 串台;服务端用 `AtomicBool`
  同一时刻只接纳一个客户端(token 路径在抢槽位前校验,错误 token 先被拒),断开后槽位释放、可被再次接管(顺序复用)。多客户端多路复用列入后续增强。
- **客户端 `Browser::connect(ws)` / `connect_with(ws, opts)`**:经 `transport::ws_connect` 建 ws → `Connection::from_ws` → 与 launch 共用
  `init_session`(`Browser.enable` + UA 掩码;Juggler 端幂等,故接管已运行浏览器安全)。**接管语义**:不启动子进程;`quit()` 只断开本地连接、
  **不关远端浏览器**;要真正关远端用 `close_remote()`,关服务端 + 浏览器用 `server.stop()`。
- **生命周期修复(关键)**:`Tab` 的事件泵 `tokio::spawn(pump(events, conn.clone(), …))` 既**持有** `Connection` 又**订阅**其事件广播,
  形成自环 → 所有句柄 drop 后泵仍 park 在 `recv()`、底层连接永不析构(attach 模式下 ws 永不关闭、服务端槽位不释放,实测重连卡 ~30s)。
  解法:`TabCore` 存事件泵的 `AbortHandle`,`impl Drop for TabCore`(最后一个 `Tab` 克隆析构)`abort()` 泵,断开自环(实测重连降到 ~100ms)。
- 依赖:`tokio-tungstenite`(由 dev-dep 提为正式依赖;仅 `ws://`,未启用 TLS feature)。完整自验证见 `examples/ws_connect`。

## 吐环境深化:指纹补环境 + 反 hook + 签名定位 + 一键导出(里程碑 30)

在通用吐环境(里程碑 20:指定/吐全/只吐关键)之上做深,形成差异化护城河。核心思路:**配方单一来源**——
canvas/webgl/audio 的采集配方 `assets/fp_recipes.js` 同时被「探针采集」与「verify 重算」内联,确保
"录制时的配方 == 回放/重算时的配方",指纹才可在 Node 补环境里被忠实回放并逐字段对齐。

- **canvas / webgl / audioContext 指纹补环境**:
  - 采集(探针 `collectSeed`):`__fpCanvas`(固定 2D 绘制→`toDataURL`)、`__fpWebGL`(curated `getParameter`
    枚举 + `WEBGL_debug_renderer_info` unmasked vendor/renderer + `getSupportedExtensions`)、`__fpAudioAsync`
    (`OfflineAudioContext(1,5000,44100)` 三角波+压缩器渲染,取 `[4500,5000)` 切片与绝对值和)。audio 异步,故探针在
    install 时预算、就绪落 `D.audioFp`,`collect` 时读取。
  - 回放(生成的 `env.js` = `assets/env_template.js` 注入 seed):`setup(sandbox)` 装一套**假 DOM**——
    `document.createElement('canvas')` 返回假 canvas(`toDataURL` 回放录制值、2D 上下文全 no-op),
    `getContext('webgl')` 返回假 gl(`getParameter`/`getExtension`/`getSupportedExtensions` 回放录制),
    `OfflineAudioContext.startRendering()` 返回把录制切片放回 `[4500,5000)` 的缓冲;另补
    navigator/screen/location/window 度量/`document.cookie`/localStorage/sessionStorage/纯 JS atob·btoa。
- **反 hook 检测**:探针 hook `Function.prototype.toString`,用 `WeakMap` 把被改写的 `fetch`/`XHR.open/send/
  setRequestHeader` 标记为 `[native code]`(toString 自身也标记),规避 `('' + fetch)` / `fetch.toString()` /
  `Function.prototype.toString.call(fetch)` 这类反 hook 检测。
- **签名 sink 调用栈通用化定位**(`signers()`):从每条命中签名请求的 writer 栈里取**最靠上的 http(s) 脚本帧**
  (即真正调用 fetch/XHR 写签名的脚本),按文件聚合计数、频次降序。兼容 Firefox(`fn@url:line:col`)与
  Chrome(`at fn (url:line:col)`)两种栈格式;不依赖站点名(从 bdms 专用 → 任意站点通用)。
- **一键导出工程**(`export_project(dir, scope)`):吐出**零依赖、可直接 `node` 运行**的补环境工程——
  `env.js`(补环境+指纹回放)/`index.js`(`createEnv`/`loadScript`/`run` 沙箱入口)/`demo.js`(纯算签名示例:加载
  `signer/` 下脚本并枚举签名函数)/`verify.js`(回放自检)/`package.json`/`README.md` + `seed/signers/targets/sinks.json`
  + 空 `signer/` 目录。
- **验证(双重)**:① 库内 `verify` 同构双跑——navigator/screen/location 浏览器与 Node 各实时读(强一致);
  canvas/webgl/audio 因 Camoufox 可能逐次给 canvas 加噪,改比对**「录制值 vs Node 回放值」**(`fill_recorded_fp`
  把浏览器侧指纹换成录制值,与 Node 回放同源对齐)。② 导出工程自带 `node verify.js`——`env.js` 回放 vs `seed.json`
  录制值逐项核对。`examples/dump_env_fingerprint` 离线实测:库内 **39/39 一致**、工程 **33/33 一致**、
  `('' + fetch)` 显示 native,ALL CHECKS PASSED。

## CDP 后端(Chromium / Chrome / Edge / Electron,里程碑 43/44;**v0.2.0 起为默认后端**)

**默认后端**(`default=["cdp"]`):驱动或接管 Chromium 系浏览器(默认 Google Chrome)与内置 CDP 的 Electron 桌面应用;
Camoufox/Juggler 反检测后端及其全部高层能力改为 `--features camoufox` 一行开启。
**核心红利**:CDP 线消息(`id`/`method`/`params`/`sessionId` + `result`/`error`/事件)与 Juggler **同构**,
直接复用 [`protocol::Connection::from_ws`] 的请求/响应/事件机制,**不重写传输层**,只换方法名与对象模型。

### 浏览器路径探测(`cdp/locate.rs`,默认 Google Chrome,对标 DrissionPage `get_chrome_path`)

为“Windows 稳定支持 + 默认支持谷歌浏览器”,`ChromiumBrowser::launch` 自动定位本机浏览器,探测优先级:

1. **环境变量** `CHROME_BIN` / `DRISSION_CHROME`(显式指定,最高优先);
2. **常见安装路径**(Chrome 优先):Windows 同时覆盖**用户级** `%LOCALAPPDATA%`(免管理员安装,极常见)与系统级
   `%PROGRAMFILES%` / `%PROGRAMFILES(X86)%` / `%PROGRAMW6432%`,含 Chrome/Chromium/Brave/Edge;macOS 查 `/Applications/*.app`;Linux 查 `/usr/bin`、`/opt`、`/snap` 常见位置;
3. **Windows 注册表** `SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths\chrome.exe` 的默认值(`HKEY_CURRENT_USER` 优先,
   再 `HKEY_LOCAL_MACHINE`)——这是 Windows 应用注册自身可执行路径的规范位置,也是 DrissionPage 的核心做法;经 `windows-sys`
   的 `RegOpenKeyExW`/`RegQueryValueExW`/`RegCloseKey`(新增 `Win32_System_Registry` feature);
4. 系统 **`PATH`** 扫描(`std::env::split_paths` 正确处理 Windows `;` 与 unix `:`)。

**为何对标 DP 重写**:旧实现仅 3 个硬编码 Program Files 路径 + 只认 `CHROME_BIN`,免管理员用户级安装(装到
`%LOCALAPPDATA%`)、非默认盘、PATH 注册的 Chrome 都探测不到,是 Windows 上“找不到浏览器”的高发原因。
**便捷方法**:`ChromiumBrowser::launch_with(path, headless)` 指定可执行文件、`ChromiumBrowser::find_chrome()` /
`cdp::chrome_path()` 诊断。像素/注册表重活在 `#[cfg(windows)]` 分支,纯函数(`scan_dirs`/`install_candidates` 等)带跨平台单测。

- **组件化**(`src/cdp/`):`core`(CdpCore 共享内核)/ `browser` / `tab` / `element` / `listener` / `interceptor`,
  职责单一、各自带单测;`CdpCore` 被同一标签的 `Tab`/`Element`/监听/拦截句柄经 `Arc` 共享(同 Juggler `TabCore`)。
- **对象模型差异(与 Juggler 的关键不同)**:
  - 元素句柄由 `objectId` 标识,操作走 `Runtime.callFunctionOn`,函数里 **`this` 即元素**
    (`function(){ return this.innerText; }`),而 Juggler 的 `callFunction` 把节点当**首参**(`(node)=>...`)。
  - 查元素:`Runtime.evaluate{returnByValue:false}` 取 `result.objectId`;查多个用 `Array.from(querySelectorAll)`
    取数组句柄再 `Runtime.getProperties{ownProperties}` 展开(对标 Juggler 的 `getObjectProperties`)。
- **原生可信输入**:点击/拖拽走 `Input.dispatchMouseEvent`(`button`+`buttons`+`clickCount`,`isTrusted=true`);
  键盘走 `Input.dispatchKeyEvent`(`keyDown` 带 `text` 即敲入字符)+ `Input.insertText`(快速整段)。
  拟人轨迹复用 minimum-jerk + `fire_session` 不等往返的密集采样(同里程碑 32 思路)。
- **网络监听**(`tab.listen()`):**原生 `Network` 域**事件(`requestWillBeSent`/`responseReceived`/`loadingFinished`)
  按 `requestId` 聚合 + `Network.getResponseBody`(CDP 下稳定,含同源/跨域;`base64Encoded` 区分文本/二进制)。
  异于 Juggler 后端因 `getResponseBody` 不稳才走页面 hook——CDP 原生即稳,**不污染页面**、覆盖全资源类型。
- **请求拦截**(`tab.intercept()`):**`Fetch` 域** `requestPaused`,匹配过滤的交用户(`resume`/`resume_with`/`fulfill`/
  `abort`),不匹配的自动 `Fetch.continueRequest`(避免页面卡死);决策方法消费 `self`,类型保证每请求只决策一次。
- **数据类型跨后端复用**:`DataPacket`/`RequestData`/`ResponseData`/`ListenFilter`/`ResumeOptions`/`Keys`/`KeyInput`
  是后端无关数据,CDP 直接复用 `crate::browser` 的定义,两后端 API 一致、用户心智统一。
- **资源回收(自环规避)**:监听/拦截后台任务与 `CdpInterceptedRequest` **只持 `Connection`+`session_id`**(不持
  `CdpCore`),`CdpCore` 里存任务 `AbortHandle`,`impl Drop for CdpCore` 在最后一个句柄析构时 `abort`——否则
  "任务/排队请求 → 反持 core → core 存任务句柄"成环,标签 drop 后任务永 park、core 永不析构(同 Juggler `TabCore` Drop 教训)。
- **验证**:`examples/cdp_advanced`(进程内极简 HTTP 服务 + 本机 Chrome headless,完全离线)端到端自验证元素句柄/
  拟人输入/可信点击(`isTrusted=true`)/网络监听(抓到 `/api/data` 200 + 响应体)/拦截(`fulfill` 伪造响应页面实收),
  **ALL CHECKS PASSED**;**无新依赖**。**v0.2.0 起 CDP 为默认 feature**(`default=["cdp"]`),`cdp_demo`/`cdp_advanced`
  示例默认即可运行;Camoufox 系示例需 `--features camoufox`。

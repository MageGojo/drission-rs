# 主动逆向能力补全(设计与里程碑)

> 定位:把 `drission-rs` 从「能自动化(操作浏览器 + 反检测过盾 + 补环境/纯算 + 验证码)」升级到
> **「能像逆向工程师一样在浏览器里下断点、抓调用栈、偷密钥、defuse 反调试、抓包重放」**。
> 恢复上下文顺序:`docs/需求.md` → `docs/设计.md` → `docs/进度.md` → `docs/第三梯队.md` → 本文件。

## 背景:逆向实战链路 vs 现有能力

一个 web 逆向工程师的标准链路是:**拿源码 → 定位 sink → 断点跟栈 → 偷密钥/复现 → 重放验证**。
核验后(全库 grep)结论很清晰——库此前几乎全压在「补环境 / 纯算复现」这一环,**最前面的「主动定位」环节基本空白**:

- `Debugger.*` / `DOMDebugger` / `setXHRBreakpoint` / `getScriptSource` / `searchInContent` → **此前全库 0 命中**(CDP 调试器域完全没接);
- `blackbox` / `setSkipAllPauses` → **0**(无反「无限 debugger」);
- 但 `expose_function`(`Runtime.addBinding`)+ `Page.addScriptToEvaluateOnNewDocument` 已有 → **hook 回传的基建现成**,只差封装。

也就是说:库能「像人一样操作浏览器并补环境」,但还不能「像逆向工程师一样在浏览器里下断点、抓调用栈、偷密钥」。补上这条线,才是真正「让 web 逆向更简单」。

## 六刀(按 ROI 排序实现,全部基于现有 `CdpCore.send` + 事件广播 + addBinding,不引重依赖)

## 关键设计:复用既有内核,薄封装、零重依赖

所有能力都是在 [`CdpCore::send`](../src/cdp/core.rs) + [`Connection::subscribe`](../src/protocol/connection.rs)
事件广播之上的**薄封装**,严格沿用既有「**`tab.xxx()` 句柄 → 返回带 `AbortHandle` 的守卫 + 后台 pump 订阅事件按 `sessionId` 过滤**」模式(同 `tab.listen()`/`tab.console()`/`expose_function`):

- **① 调试器** `src/cdp/debugger.rs`:`Debugger.enable` + `DOMDebugger.setXHRBreakpoint`/`setEventListenerBreakpoint`;
  `wait_paused` 订阅 `Debugger.paused` 取 `callFrames`(函数名/scriptId/行列 + scopeChain);
  `PausedStack` 持 `Arc<CdpCore>`,`eval`=`Debugger.evaluateOnCallFrame`、`locals`=`Runtime.getProperties` 展开局部作用域、`resume`/`step_*`。
- **② 脚本** `src/cdp/scripts.rs`:`Debugger.enable` 触发 `scriptParsed` 回灌(含已解析脚本),
  `Debugger.getScriptSource` dump、`Debugger.searchInContent` 全文搜索;美化器 `beautify_js` 为纯 Rust、
  **字符串/模板串/正则/注释感知**(不破坏字面量)。
- **③ Hook** `src/cdp/hook.rs`:复用 `Runtime.addBinding` + `addScriptToEvaluateOnNewDocument` + `bindingCalled` pump;
  注入的 wrapper 全程 try/catch(**绝不弄坏页面**),`safeSerialize` 把 ArrayBuffer/TypedArray → base64、CryptoKey → 元信息;
  `with_stack` 用 `new Error().stack`。
- **④ 反反调试**(`src/cdp/debugger.rs`)**两条路,互补**:
  - **注入式 `tab.anti_anti_debug()`**:导航前注入脚本 —— 包 `setInterval`/`setTimeout`(handler 源码含 `debugger` 则换 no-op)、
    致残 `Function`/`Function.prototype.constructor`/`eval`(函数体/eval 串里的 `\bdebugger\b` 剔除)。自包含、**不依赖 Debugger 域**。
    **局限(真站实测)**:`debugger` 若藏在「被调度回调**间接调用**的具名函数」或「已解析的内联脚本」里,注入改不到。
  - **CDP 原生通杀 `tab.debugger().set_skip_all_pauses(true)`**(`Debugger.setSkipAllPauses`):调试器对**一切暂停**视而不见
    ——`debugger` 语句、**计时型反调试(checkPerformance)**、间接/`eval` 生成的 `debugger`、甚至你自己的断点**全部跳过**。
    **这是强反调试站的正解**;想用断点定位时再 `set_skip_all_pauses(false)`。`count_pauses(window, auto_resume)` 可量化反调试强度(对照 defuse 前后)。
- **⑤ 重放** `SessionPage::replay`(`src/session/mod.rs`):从 `DataPacket` 取 method/url/headers/body,过滤逐跳头
  (host/content-length/connection 等)与可选 cookie,链式 `set`/`set_header`/`url`/`body` 覆盖后走既有
  cookie/重定向/`impersonate`(wreq TLS/JA3)发送循环——**复现签名后立刻验真**。
- **⑥ WASM** `tab.scripts().dump_wasm`:`scriptParsed` 里 `scriptLanguage=="WebAssembly"` 的脚本经 `getScriptSource`
  取 `bytecode`(base64)落 `.wasm`,提示用 `wasm2wat`/wabt 反汇编(全 wat 反汇编不进库,避免重依赖)。

## 取舍 / 边界(说实话)

- **不替你理解业务陷阱**:key 藏在 `newSign[4:20]`、地址在 `watchInfo` 不在 `sortedItems`、token 按 IP 绑定——这些仍要人去啃;
- **强混淆/VMP、行为风控评分**库帮不上根本忙;
- **`Debugger`/`Runtime.enable` 有反检测代价**:开调试器域会被部分站点的「CDP 探测」识别,故①②③④的调试/ hook 能力**默认不开**,
  按需调用、用完即停(守卫 drop 自动收);极致反检测场景(过盾那条线)继续走「不开 Runtime.enable」的既有路径。
- **crypto tap 偷不到不可导出的 key**:`crypto.subtle` 的 `CryptoKey` 若 `extractable=false`,只能回传算法/类型元信息 + 明文/密文数据,key 本身偷不到(这是 WebCrypto 的设计);`CryptoJS` 路线则通常能直接拿到 key 串。

## 真站实战(`mh.yichengwlkj.com` / `api.rrmj.plus`,人人视频)

实测既验证了能力、也暴露并补齐了缺口(`re_verify` / `re_rrmj` / `re_rrmj_replay`):

- **④ 反调试**:① 调用栈定位到 `checkPerformance`(空 URL=eval 生成)**计时型反调试**,8s 暂停 7 次。
  注入式 `anti_anti_debug` 对这种「藏在间接调用/已解析脚本里的 `debugger`」**改不到** → 因此新增 **CDP 原生
  `set_skip_all_pauses(true)` 通杀**:7 次 → **0 次**,页面照常加载,可干净调试/dump。
- **③ Hook**:`xhr()` 抓到 `setRequestHeader('x-ca-sign', …)` 全部值 + 栈直指签名 chunk(`2112-….js:1:118656`);
  用 `open→setRequestHeader→send` 序列**重建出每个请求的完整 12 个头**(`token/cv/ct/t/deviceid/x-ca-sign…`),
  补上了 **CDP `requestWillBeSent.request.headers` 漏头**(只有 accept/x-ca-sign)的坑。
- **⑤ 重放 + AES 解密**:`session.replay(&pkt)`(Chrome TLS 指纹)重放签名请求 **status=200**,响应 **AES-128-ECB
  (key `3b744389882a4067`)解出真实剧集 JSON**(`"title":"世界的主人"…`)——**重放验真端到端打通**。
- **复现的边界(诚实)**:`x-ca-sign` = `base64(HMAC-SHA256(签名串, "ES513W0B1CsdUrR13Qk5EgDAKPeeKZY"))`,
  算法骨架与 key、全部入参(12 个头)均已拿到;但**签名串的确切拼法是自定义混淆**——`x-ca-sign` 字符串名在源码里
  动态构造(② 字面 grep 命中 0)、HMAC 是 webpack 内置(非全局 `CryptoJS`,③ `crypto_js` 偷不到)、多候选 brute 未命中。
  **要拿确切拼法须用 ① 在 hmac 处下断点读入参 / 精读 chunk 2112**——这正是「工具给定位、最后一公里靠人」的业务陷阱,
  不属算法/工具缺陷。

## 真站实战(YouTube base.js,`re_yt_play`:sig + n 端到端、下载验真)

把六刀用到「业界最硬之一」的 YouTube 播放器签名上,**端到端产出真实能播的 `googlevideo` 直链并下载 + ffprobe 验真**:

- **统一 VM 认知**:现代 base.js(2024+)把 **签名(sig)与 n 参数解扰统一进同一个 opcode VM**——经典
  `a=a.split("")…return a.join("")` 已不存在,纯静态正则抠不出干净件。看清后发现一个「**万能直链构建器**」:
  `y2=function(m,Z="",J=""){m=new g.g7(m,!0);m.set("alr","yes");J&&(J=Wv(65,2902,Wv(3,4369,J)),m[c[12]](Z,$8(21,6101,J)));return m}`
  —— `y2(url, sp, s)` 返回的 URL 对象:**n 已由 `g.g7(…,!0)` 解扰、签名 `s` 已由 `Wv` 解密并 `set`**;序列化方法
  `.JK()` 名字会轮换 → **运行时探测原型上的无参方法、取返回 `http(s)://…` 的那个**(对名字轮换免疫)。
- **捕获手法(①)**:`y2`/`g7` 都是模块闭包私有件(全局取不到)。在 base.js 求值时断在 `cs6` 赋值处(列 0,必命中),
  用 **`expose_as_global` 在断点帧作用域里造一个闭包函数挂到 `window`** —— 它闭包住模块作用域里的 `y2`,resume 后随处可调
  (**call 时才解析,无顺序/版本依赖**)。同款手法同时暴露 `__ytNsig`(闭包 `g.g7`,单独展示 n 变换)。
- **干净环境 + 就地解析**:`tab.scripts()` 定位时 `list()` 会 `setSkipAllPauses(true)` 屏蔽断点 → 下断点前必须
  `set_skip_all_pauses(false)`;**解扰类 / `y2` 名 / 行号全从活 base.js 就地解析**——实测两次跑名字不同
  (`g.iE`/`Ah` vs `g.g7`/`y2`),证 base.js 在轮换、**零硬编码**才不挂。
- **端到端 + 验真**:从 `ytInitialPlayerResponse` 取 progressive(itag 18,`signatureCipher`)→ `__ytUrl` 自算终极直链
  (n 乱序→解扰、sig 解密 set、确定性双跑一致)→ **reqwest `Range` 真实下载 HTTP 206 / `video/mp4` / 4 MB 落盘**
  → **ffprobe 验真**(本地文件 + 直连 URL 双重):`h264 640x360 + aac 44100`,真实时长(279.66s / 213.04s)吻合。
  另写了 **`pot` 透传回退**(我方链没带、播放器真实请求带了就借用——`pot` 是会话 token,同 cookie 透传,非「复现」;itag 18 本次无需)。
- **诚实边界**:`n` 解扰的 opcode VM 本身(`Wv`/`pV` 派发器)未逐条反汇编成纯算 —— 走的是**运行时抠件复用**
  (把页面自带的 `y2` 当 oracle),而非脱离浏览器的纯 JS/Rust 重写。要纯算还需精读 VM(后续可选);
  但「自算直链 + 真实下载验真」这条端到端已完整闭环。

## 纯协议 · 脱浏览器(`re_yt_protocol`:reqwest + 内嵌 QuickJS,零 Chrome)

`re_yt_play` 仍要开浏览器抠 oracle;`re_yt_protocol` 把它做成**完全不开浏览器**的协议版(对标 yt-dlp 的思路):

- **为什么不逐 opcode 重写**:nsig/sig 是同一套 opcode VM(`g7.get→pV(8,6793,this)`、`y2` 的 sig→`Wv(65,2902,…)`;
  `pV`/`O9`/`Wv` 互递归派发器 + `c` 字符表)。逐条反汇编进 Rust 既随 base.js 每几小时轮换而碎、又无收益——
  **yt-dlp 也是把 base.js 喂进自带 JS 解释器跑、不重写 VM**。故走「整份 base.js 进 JS 引擎」。
- **管线**:`reqwest` 抓 watch 页(拿 `ytInitialPlayerResponse` + `/s/player/<ver>/…/base.js` URL)→ 抓 base.js →
  **内嵌 QuickJS(`signer`/rquickjs)补环境跑整份 base.js** → 顶部注入的惰性 `__ytUrl/__ytNsig` 自算 n/sig →
  拼直链 → `reqwest` Range 下载 + ffprobe(本地 + 直连 URL)。
- **补环境(关键)**:base.js 是浏览器脚本(`var window=this` + 大量 DOM/CSS/定时器初始化,`g.g7` 类在 ~6789 行才定义)。
  把 `document/navigator/location/console/atob/TextEncoder/Intl…` 桩装到 `globalThis`,base.js 以 **sloppy 全局**求值
  (`var _yt_player`→全局、`var window=this`→globalThis 带桩);顶部注入 `__ytStash(g)` + 惰性 oracle(闭包 `y2`/`g.g7`,
  call 时解析)——即便后续 DOM init 抛错,核心已定义、oracle 已 stash,仍可调用。
- **QuickJS 三个坑(已解)**:① **rquickjs `eval` 默认 `strict=true`**(禁 `with`)→ 用 `eval_with_options{strict:false}`;
  ② **`with(Proxy)` 包裹整份 base.js 触发 QuickJS 编译器 bug `inconsistent stack size`** → 弃 `with`,改 globalThis 桩 + 全局求值;
  ③ **QuickJS 无 `Intl`,且 base.js 顶层就调 `Intl.NumberFormat.supportedLocalesOf(...)`(在 `g.g7` 定义之前)** →
  必须补 `Intl` 并给各构造器加**静态 `supportedLocalesOf`**,否则在定义 `g.g7` 前就抛、命名空间拿不到。
- **验证**:2 个视频(`dQw4w9WgXcQ` / `7349tcyyE-c`)纯算 itag18 直链均 **HTTP 206 下载 + ffprobe 验真**
  (h264 640x360 + aac,时长 213.04s / 279.66s 吻合)。
- **诚实边界**:脱的是**浏览器**、不是 **JS 引擎**——VM 执行交给 QuickJS,不是纯 Rust 重写(与 yt-dlp 同)。
  对混淆/名字轮换免疫(`y2`/解扰类/行号全就地解析);真做逐 opcode 纯 Rust 化是另一个量级、且收益低。

## 通用 CLI 下载器(`yt_dl`:选清晰度 / 合并 adaptive 高清 / 进度条 / pot,`--features signer,cdp`)

把 `re_yt_protocol` 的纯算引擎抽成共用模块 `yt_engine.rs`(fetch / InnerTube / 格式 / QuickJS solver / 分块下载 / 合并 / pot),`yt_dl.rs` 是薄壳 CLI:

```bash
cargo run --example yt_dl --features signer,cdp -- <URL|videoId> [-q best|2160|1080|720|480|audio|itag:N] \
  [-o name] [--list] [-g|--get-url] [--client tv|web] [--audio-only] [--no-merge] [--pot TOKEN] [--browser-pot]
```

- **HD 分轨从哪来**:watch 页 `ytInitialPlayerResponse` 的 adaptive 被剥了 url(`withUrl:0`),只有 progressive(itag18)带 cipher。
  要全部高清分轨须走 **InnerTube `/youtubei/v1/player`**。**关键:默认用 `TVHTML5` 客户端——免 pot 即返回全部 adaptive(实测 27 路、到 2160p vp9/av1),全带 signatureCipher**,正好喂 QuickJS 算 sig/n;`ios/android` 现需 attestation 拿不到,`WEB` 不带 pot 不给 adaptive url。
- **选清晰度 + 合并**:`select_plan` 按高度/帧率/码率挑视频 + 最佳音频 → QuickJS 各算直链 → 分别下载 → **ffmpeg `-c copy` 合并**(mp4/webm/mkv 按 codec 选容器);progressive/`audio` 走单文件。
- **【头号坑·限速】googlevideo 对「不带 `Range` 的整文件 GET」节流到接近实时码率**:实测同一条直链、同一个已解扰 n——
  **整发 GET 仅 ~130KB/s(HTTP 200);带 `Range: bytes=a-b` 分块则 ~9–20MB/s(HTTP 206)**。不是签名/n 问题(同链 curl 同样慢/快)。
  故 `download_with_progress` 必须**逐块 Range 下载**(默认 10MB/块,`CHUNK_MB` 可调),与 yt-dlp 同思路。这是最有价值的通用修复。
- **【环境坑·403】走 WARP / 轮换代理时 adaptive 下载间歇 403**:googlevideo 把铸链时所见**出口 IP 签进 `ip=`(进 `sparams`)**,下载出口须与之一致。
  直连时两者天然相同;但 **WARP 会给 `youtube.com`(铸链)与 `googlevideo.com`(下载)分配/轮换不同出口 IP** → `ip=` 不符 → 403。
  **铁证**:对 reqwest 刚铸链(已 403)的同一条 url,此刻 `curl` 默认/`-4`/`-6` **也全 403**(连 curl 都拉不动)→ 纯属代理出口轮换,与代码/签名/Range 无关。

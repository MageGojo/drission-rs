# 长监听 + 滑动驱动 设计

> 目标:在一个抖音视频页上,**持续**监听 `aweme/v1/web/aweme/detail` 这类接口;
> 通过程序模拟「下滑/下一个视频」让页面 JS 不断发出**下一条**视频的带签名请求,
> 从而连续抓到每个视频的参数(`a_bogus` / `verifyFp` / `fp` / `webid` / `msToken` 等)。
> **不破坏**现有的单次抓取(`listen_wait` 抓第一条)。

## 1. 现状回顾(为什么不冲突)

监听用的是**页面内常驻 hook**(`window.__drission_net_q`):
- `listen_xhr/listen_start` 通过 `Page.setInitScripts` + 立即注入,把 fetch/XHR 包一层,
  匹配 URL 的请求连同响应体 push 进页面队列(队列在**同一文档**内常驻)。
- SPA 内部的后续请求(切视频走 `pushState`,文档不变)会**持续入队**。
- `listen_wait` / `listen_next`:每次把页面队列 drain 进 Rust 端 `listen_buf`,再 pop 一个。

所以"长监听"本质 = **持续把队列搬运到 Rust 端 + 程序驱动页面加载下一条**。
与单次抓取是同一套 hook、同一个 `listen_buf`,只是消费方式不同 → 天然不冲突。

## 2. 两块能力

### 2.1 持续消费(后台抽取 + 流式 API)

问题:`listen_wait` 只在被调用时才 drain;长会话里两次调用之间页面队列可能涨满
(hook 内有上限,默认会丢老包)。长监听需要**不丢包**。

方案:**可选**的后台抽取任务 + 流式句柄,且与单次模式互斥地共用一个缓冲:

- `TabCore` 增加:
  - `bg_active: AtomicBool` —— 是否已开启后台抽取。
  - `listen_bg: Mutex<Option<JoinHandle<()>>>` —— 后台任务句柄。
- `Tab::listen_forever()`:要求 hook 已激活;幂等地 spawn 一个后台任务,
  每 ~120ms `evaluate(DRAIN_JS)` 把页面队列搬进 `listen_buf`。置 `bg_active=true`。
- `Tab::listen_stream() -> ListenStream`:确保后台抽取已开,返回流式句柄。
  - `ListenStream::next()`:等待并取下一个包(默认超时)。
  - `ListenStream::next_timeout(d)`:带自定义超时,超时返回 `None`(用于"滑一下再等")。
- **防竞争**:`drain_into_buffer` 在 `bg_active==true` 时变为 **no-op**
  ——drain 只由后台任务做,`listen_wait`/`ListenStream` 都只从 `listen_buf` pop。
  这样后台抽取与按需 drain 不会同时 `DRAIN_JS`(`DRAIN_JS` 是"取走并清空",并发会瓜分数据)。
- `listen_stop()`:先 `listen_active=false` 让后台循环退出 → abort 句柄 → `bg_active=false`
  → 卸载 hook、清空缓冲。

**关键:单次模式(只调 `listen_wait`,不开 `listen_stream`)行为完全不变**
——`bg_active` 为假,`drain_into_buffer` 仍按需 drain。现有 `douyin_listen` 示例零改动。

### 2.2 滑动 / 下一个视频(输入驱动)

抖音网页版切到下一个视频:**方向键 ↓** 或 **鼠标滚轮下滑** 都行,都会触发新的
`aweme/detail` 请求。Juggler 输入方法(取自 Playwright Firefox 实现):

- `Page.dispatchKeyEvent`:`{type:"keydown"|"keyup", key, code, keyCode, location, repeat, text}`
- `Page.dispatchWheelEvent`:`{x, y, deltaX, deltaY, deltaZ:0, modifiers}`

新增 `Tab` 方法:
- `press_key(key)`:发 keydown+keyup;内置常用键映射(ArrowDown=40/`ArrowDown` 等)。
  → 抖音切下一个视频用 `press_key("ArrowDown")`,**最简单、无需坐标**。
- `wheel(dx, dy)` / `wheel_at(x, y, dx, dy)`:在视口中心(或指定点)派发滚轮。备选触发。
- `scroll_by(x, y)`:`window.scrollBy`,普通页面滚动(对抖音 feed 不一定切视频,通用用途)。

## 3. 典型用法(示例 `douyin_listen_long.rs`)

```text
listen_xhr(["aweme/v1/web/aweme/detail"])   // 导航前开 hook
get(短链)                                    // 第一条 detail 已入页面队列
stream = listen_stream()                      // 开后台抽取 + 拿流
loop 直到抓够 N 个:
    p = stream.next_timeout(15s)
    若拿到新 aweme_id:打印 a_bogus/verifyFp/... 等参数;got++
    press_key("ArrowDown")                     // 滑到下一个视频
    sleep 2s                                    // 等下一条 detail 发出
```

去重:按 URL 里的 `aweme_id` 去重(同一视频可能重复发 detail)。

## 4. 边界与权衡

- 焦点:刚导航完焦点在 body,`↓` 能到抖音的 document keydown 处理器;若焦点落在评论/搜索框
  则方向键失效(本流程不聚焦输入框,无此问题)。
- opaque 跨域响应体页面层读不到(沿用现有 hook 局限);抖音 detail 是同源,无影响。
- 后台抽取频率 120ms 为经验值;可按需要调。`listen_buf` 为无界 `VecDeque`,长会话内存随
  抓取量增长(够用;如需可加容量上限淘汰旧包)。
- `press_key` 未知键:只发 key 名(`text` 留空),不保证生效;常用导航键已覆盖。

# drission-rs 差异化路线

## 核心判断

不要把 drission-rs 做成"Rust 版 Python DrissionPage"。Python 用户不会因为 API 长得像就放弃成熟业务栈,所以 DP 风格 API 只能是上手入口,不能是产品护城河。

drission-rs 真正该做的是:浏览器自动化运行时 + 账号/Profile 身份治理 sidecar。Python 继续写业务逻辑,Rust 接管更适合系统层做的事:账号池准入、profile 租约、续租、释放、失败判责、冷却、修复态、熔断、风险建议、跨轮门禁和审计流水。

## 不继续卷的方向

- 不长期追求 DrissionPage API 逐项 1:1。缺常用能力可以补,但不要让路线图被 Python API 表牵着走。
- 不把"请用户迁移到 Rust"作为采用前提。迁移成本太高,收益也不够直接。
- 不空喊"更快"。浏览器自动化瓶颈多数在浏览器、网络、站点风控和账号质量,不是语言本身。
- 不承诺通用过盾。应该输出可验证的诊断、动作、风险和证据,而不是玄学保证。

## 要做成护城河的能力

1. **运行时账号治理协议**

   `identity-job run` 包住旧 Python/Node/shell 脚本,在业务启动前完成 sweep、validate、runtime-risk-gate、gate、select,业务结束后自动 release。脚本只管写业务结果,Rust sidecar 负责账号/Profile 生命周期。

2. **失败原因变成下一轮调度信号**

   子进程可写 `{"status":"failed","reason":"rate_limited"}`。policy 的 `job.failureReasonRules.rate_limited` 不只决定本次 `cooldownSeconds/nextState`,还可以决定 `recommendedAction`、`runtimeRiskSeverity`、`nextSuggestedLimit`、`nextSuggestedDesiredConcurrency` 和 `runtimeRiskCooldownSeconds`。这会生成 `suppressUntilUnixSeconds`,下一轮在领取 profile 前按精确截止时间拦截或降速。

3. **Profile lease 与 release ledger**

   每个账号/Profile 都有 runtime lease、lease renewal、timeout watchdog、release ledger、health 判责和 sweep 清理。目标是任何 worker 崩溃、脚本卡死、限流、风控、失败冷却都可追溯、可恢复、可回放。

4. **身份一致性与账号池准入**

   `identity_report`、`identity-pool`、`identity-drift`、`identity-lifecycle` 把 UA、Client Hints、WebGL、canvas、时区、语言、screen、profile 稳定性变成评分、ledger 和 action queue。重点不是"随机指纹",而是证明这一批账号看起来不像同一套模板。

5. **机器可执行 action queue**

   所有诊断都要能落成动作: quarantine、repair、restore、rotate、capacity disperse、pause_pool、pause_failure_reason、reduce_concurrency。AI Agent 或调度器应读结构化动作,而不是读自然语言建议。

6. **跨语言控制面**

   CLI/MCP/env/result JSON 是主接口。Rust 不是替换 Python,而是变成 Python 业务旁边的控制面。后续可以补 Python 小包,但它只应薄封装 `drs`,不要把核心治理逻辑搬回 Python。

7. **审计和复盘**

   每次为什么跑、为什么不跑、哪个账号被拿走、哪个原因导致冷却、哪条风险挡住下一轮,都应该能从 NDJSON ledger 和 job report 还原。规模化业务最缺的是这个,不是又一个 `ele().click()`。

## 当前已经落地的特殊能力

- `identity-job run` sidecar:自动 sweep/validate/runtime-risk-gate/gate/select/child/release。
- per-asset 子进程池:旧的一进程一账号 Python 脚本不需要改成 Rust。
- runtime lease renewal 和 child timeout watchdog。
- 全局失败熔断 `maxFailedAssets` 与按失败原因熔断 `maxFailedAssetsPerReason`。
- 子进程结果协议 `DRS_IDENTITY_RESULT_OUT`。
- `failureReasonRules` 同时控制 release 冷却/状态迁移和 runtime risk。
- `runtimeRiskOut` / `runtimeRiskLedgers` 跨轮风险流水。
- `runtimeRiskCooldownSeconds` 生成 `suppressUntilUnixSeconds`;下一轮即使超出普通窗口,只要抑制未到期也会在领取 profile 前停止。
- `identity-job run` explain 模式:响应内置 `explain.stageDecisions[]` / `explain.assetDecisions[]`,并可用 `--explain-out` 或 policy `job.explainOut` 单独落盘,解释每个阶段和账号/Profile 为什么通过、阻断、选中、失败或释放。
- `identity-job run` policy preset:内置 `publish_conservative`、`login_sensitive`、`scrape_aggressive`,可通过 `--job-preset` 或 policy `job.preset` 先选择业务风险模型,再局部覆盖阈值。
- Python sidecar helper: `python/drission_sidecar` 只负责读取 `DRS_IDENTITY_ASSET_JSON`、访问 profile/account 字段、原子写 `DRS_IDENTITY_RESULT_OUT`,让旧 Python 脚本用 `succeeded()` / `failed(reason=...)` / `cancelled()` 接入治理协议,而不是迁移浏览器业务逻辑。
- `identity-ledger query`:聚合 runtime release ledger 与 runtime risk ledger,输出状态统计、失败原因排行、账号/Profile 失败排行、risk action/severity、仍生效的 `suppressUntilUnixSeconds`,回答"过去一段时间什么原因挡住了多少次、哪些账号最常失败、下一轮是否还应被 suppression 拦住"。
- `identity-ledger explain`:按 job/worker/reason/account/profile/identity/label/profileDir/leaseId 解释单个目标的 `decision`、`blockingReasons[]`、`nextRunnableUnixSeconds`、active suppression/cooldown 和对应 release/risk 证据,回答"这个账号或这个失败原因为什么现在不能跑,什么时候能再跑"。
- `identity-ledger compact`:把长期 runtime release/risk NDJSON 压成 `identity_ledger_compact` JSON 摘要,保留 active suppression、资产级成功/失败统计、失败原因分布、risk action/severity、最近证据和 `compactedThroughUnixSeconds`;支持 `--checkpoint-in/--checkpoint-out` 用 source byte offset 只读新增 tail,源文件轮转/截断时标记 reset。
- `identity-ledger dashboard`:在 compact 摘要上生成 `identity_ledger_dashboard` JSON 和可选 HTML,把 status、recommendedAction、失败率、active suppression、top assets、top reasons 和最近证据做成人可复盘的运行态页面。

## 下一批优先级

1. **P1: compact-aware scheduler examples**

   `identity-ledger query/explain/compact/dashboard` 已能聚合、解释、压缩、展示和增量读取账本,下一步应补一个调度器例子:先读 compact/dashboard JSON 判断 run/wait/pause,再调用 `identity-job run`,最后写回 release/risk ledger。

2. **P2: site/job adapter packs**

   不写通用"过某站"承诺,而是沉淀 job policy 和 failure reason schema。例如发布类业务常见 `rate_limited/login_required/captcha/escalated_review` 应该有标准治理含义。

3. **P2: compact index hardening**

   在 checkpoint 基础上继续补 source inode/hash、压缩归档索引和异常恢复策略,让超大账本轮转更稳。

## 判断一个新功能是否值得做

值得做:

- 能让 Python 业务不迁移也获得账号/Profile 治理能力。
- 能把一次失败变成下一轮更安全的调度决策。
- 能生成可审计 ledger、可执行 action 或可验证报告。
- 能降低多 worker、多账号、多 profile 并发时的误伤和抢占。

先别做:

- 只是把 Python DrissionPage 某个方法名搬到 Rust。
- 只能在 demo 里显得高级,但不能进入调度/审计闭环。
- 需要用户重写大段旧业务才能受益。
- 只有自然语言建议,没有机器可消费字段。
